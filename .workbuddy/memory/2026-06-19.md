## 23:41 — 滚动位置恢复从 v2 重写为 v3（卡片锚点）

**问题：** 用户反馈"只实现一部分不返回顶部首页位置，并未完全实现。有的是每一条资讯里的部分链接实现了，跳转链接后，返回到当前阅读位置，不是所有都实现了。"

**根因分析（4个）：**
1. rAF 循环 `tryRestore()` 在 `visibility:hidden` 页面中可能被 webview 节流/跳过 → 3秒兜底显示时位置未恢复
2. 仅保存 scrollY 像素值 → 返回时图片加载导致页面高度变化，保存的 scrollY 指向错误位置
3. 仅监听 `click`/`pagehide` — `mousedown` 是最早的必然触发事件，未利用
4. `scrollTo(y)` 像素定位不如 `scrollIntoView` 卡片锚点定位可靠

**修复 — v3 方案（4项改进）：**

1. **`<head>` 脚本**：sessionStorage key 从 `_daily_rpt_scrollY`（纯数字）改为 `_daily_rpt_state`（JSON `{y, ci}`），同时保存像素 + 卡片 data-index

2. **四通道保存**（优先级降序）：
   - ① `mousedown` 捕获阶段 — 桌面端最早触发，导航前必执行
   - ② `touchstart` — 移动端手指按下
   - ③ `pagehide` — 页面离开前兜底
   - ④ `scroll` debounce 200ms — 日常滚动

3. **恢复策略**：
   - 策略A（优先）：卡片锚点 `scrollIntoView({block:'start', behavior:'instant'})` + `scrollBy(0,-80)` — 不受图片加载影响
   - 策略B（兜底）：`scrollTo(0, state.y)` 像素定位

4. **揭示机制**：`setTimeout(fn, 80)` 替代 rAF 循环（hidden 页面 rAF 不可靠）
   - 双触发：body 末尾立即 + `window.onload` 再次
   - 3秒兜底强制显示

**同步：** 早间+晚间自动化 prompt 中的滚动恢复代码完整替换为 v3 方案；MEMORY.md 阅读位置恢复章节重写
