## 6/22 23:07 无限循环自愈体系

用户要求："如果推送失败，必须无限循环自检，无数次循环直至收到推送为止"

### 四层无限自愈架构

**Layer 0 — main.py SelfHealRunner（进程内重试）**
- 新增 SelfHealRunner 类: 8次重试 + 指数退避(15s→300s) + 自动诊断+修复
- 退出码: 0=成功, 2=重试用尽(触发workflow级救援), 1=致命
- 5种自动诊断: empty_news/feishu_auth_fail/feishu_send_fail/network_error/exception
- 重试中动态扩大搜索范围(days_back+2@第3次, +3@第6次)
- `--self-heal` 启用, `--force` 跳过幂等

**Layer 1 — feishu_sender.py 发送保障（3层fallback）**
- send_daily_report_with_retry(): 3次重试(每次刷新token) + 指数退避
- send_plain_text_message(): 纯文本兜底（卡片失败时发送摘要文本）

**Layer 2 — Workflow级自愈（跨进程重试）**
- 早间/晚间workflow: --self-heal模式, timeout 35min
- 退出码=2→等5分钟→gh workflow run重新触发自身(最多3次)
- 递归保护: /tmp/retry_count 计数器
- 晚间workflow: 跨workflow救援(检测早间缺失→先补推)

**Layer 3 — Watchdog多通道**
- 通道1: gh workflow run CLI
- 通道2: curl GitHub API直调（gh CLI失败兜底）
